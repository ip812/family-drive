## Commands

```bash
npm run dev          # Start local dev server (Vite + Cloudflare plugin)
npm run build        # Production build
npm run preview      # Build then preview locally
npm run typecheck    # Generate wrangler types + run tsc
npm run test         # Run tests once (vitest)
npm run test:watch   # Run tests in watch mode

# Database (requires CF_ACCOUNT_ID, CF_DATABASE_ID, CF_D1_TOKEN env vars)
npm run db:generate  # Generate migration files from schema changes
npm run db:migrate   # Apply migrations to D1
npm run db:push      # Push schema directly (dev only)

npm run deploy       # Build + deploy to Cloudflare Workers
npm run cf-typegen   # Regenerate worker-configuration.d.ts from wrangler.jsonc
```

## Architecture

This is a **SPA + Hono API** deployed to Cloudflare Workers — not SSR. React Router runs in `BrowserRouter` mode (client-side only).

### Entry points
- **`workers/app.ts`** — Hono app that is the Worker entrypoint. Registers all `/api/v1/*` routes and a catch-all that serves `index.html` from R2 assets for client-side routing.
- **`app/main.tsx`** — React SPA entry, creates the `BrowserRouter` with routes for `/` and `/albums/:id`.

### Directory layout
| Dir | Purpose |
|-----|---------|
| `workers/` | Hono backend — all API route handlers live in `workers/app.ts` |
| `app/routes/` | React page components (`layout.tsx`, `home.tsx`, `albums.$id.tsx`) |
| `app/components/` | Shared React components; `ui/` holds shadcn primitives |
| `db/schemas/` | Drizzle table definitions (`albums`, `images`) — re-exported from `db/schemas.ts` |
| `http/client.ts` | Client-side fetch helpers (`getV1`, `postV1`, `deleteV1`, `uploadV1`) |
| `toasts/` | Shared `Toast` type + `ToastError` class + factory helpers (`errors.ts`, `warnings.ts`, `success.ts`). Used by both Worker and React |
| `types/` | Shared request/response TypeScript interfaces (`AlbumResponse`, `ImageResponse`, etc.) |
| `migrations/` | SQL migration files generated by drizzle-kit |

### API surface (`/api/v1`)
All routes are in `workers/app.ts` and must be registered **before** the `*` catch-all:
- `GET/POST /api/v1/albums`
- `GET/DELETE /api/v1/albums/:id`
- `GET /api/v1/albums/:id/images` — paginated (limit/offset), ordered by `takenAt DESC NULLS FIRST`
- `POST /api/v1/albums/:id/images` — multipart/form-data (`files[]` + `metadata` JSON)
- `DELETE /api/v1/albums/:albumId/images/:imageId`
- `GET /api/v1/images/*` — streams private R2 objects with immutable cache headers

### Cloudflare bindings
- **`c.env.DB`** — D1 database (use `drizzle(c.env.DB!)`)
- **`c.env.BUCKET`** — R2 bucket for file storage; keys follow `albums/{albumId}/{uuid}.{ext}`
- **`c.env.ASSETS`** — static assets binding (serves the built SPA)

### Toast / error pattern
API responses that are errors always return a `Toast` object `{ code: number, message: string }`. Throw `new ToastError(...)` inside route handlers — the `app.onError` handler converts it to the correct HTTP response. On the client, use `isToast(result)` to detect errors and `isSuccess`/`isWarning`/`isError` to classify them.

### File upload flow
Files are uploaded one at a time from `UploadButton` (sequential loop). EXIF `DateTimeOriginal` is extracted client-side via `exifr` and sent as `metadata` JSON alongside each file in the FormData. The server stores `takenAt` in D1 and the raw file in R2.

### tsconfig notes
`tsconfig.cloudflare.json` must include `db/**/*`, `http/**/*`, `toasts/**/*`, `types/**/*` in its `include` array, in addition to `app/**/*` and `workers/**/*`, otherwise cross-directory imports produce TS6307 errors.
